# End-to-End Testing Guide

This guide walks you through deploying and testing the custom trace propagator solution for GCP Cloud Run.

## Quick Start

### 1. Prerequisites

```bash
# Authenticate with Google Cloud
gcloud auth login

# Set your project
gcloud config set project YOUR_PROJECT_ID

# Install jq for JSON parsing (optional, for better output)
brew install jq  # macOS
# or
sudo apt-get install jq  # Linux
```

### 2. Get Last9 Credentials

1. Go to [Last9 Dashboard](https://app.last9.io)
2. Navigate to **Settings** > **OTLP Ingestion**
3. Copy your:
   - **OTLP Endpoint**: `<your-otlp-endpoint>` (or your specific endpoint)
   - **Authorization Header**: `Authorization=Basic <your-base64-credentials>`

### 3. Set Environment Variables

```bash
export OTLP_ENDPOINT='<your-otlp-endpoint>'
export OTLP_AUTH='Authorization=Basic YOUR_BASE64_CREDENTIALS_HERE'
```

**Important**: The `OTLP_AUTH` must include the `Authorization=` prefix.

### 4. Deploy and Test

```bash
cd /path/to/l9_otel_examples/gcp/cloud-run/nodejs

# Run full deployment and testing
./deploy-and-test.sh all
```

This will:
1. ✓ Check prerequisites (gcloud, APIs)
2. ✓ Create/update Last9 credentials secret
3. ✓ Deploy the function with custom propagator
4. ✓ Deploy the service with custom propagator
5. ✓ Run trace propagation tests
6. ✓ Show you how to verify in Last9

## Verification in Last9

### 1. Open Last9 Dashboard

Go to: https://app.last9.io

### 2. Navigate to Traces

**APM** > **Traces**

### 3. Filter by Service

Filter by either:
- Service name: `otel-test-service`
- Function name: `otel-test-function`

### 4. Find Recent Traces

Look for traces from the last few minutes (generated by the test script).

### 5. Check Span Hierarchy

Click on a trace from the **Service → Function chain** test.

**✓ Expected (with custom propagator):**

```
┌─ GET /chain (service span)                    ParentSpanId: null
│  ├─ HTTP GET (client span)                    ParentSpanId: <service-span-id>
│  │  └─ GET helloHttp (function span)          ParentSpanId: <client-span-id>
```

All spans should:
- ✅ Share the same `TraceId`
- ✅ Show proper parent-child hierarchy
- ✅ Have `ParentSpanId` references that point to spans PRESENT in the trace

**✗ Without custom propagator (broken):**

```
┌─ GET /chain (service span)                    ParentSpanId: null
├─ HTTP GET (client span)                       ParentSpanId: <service-span-id>
└─ GET helloHttp (function span)                ParentSpanId: <gcp-lb-span-id> ❌ MISSING!
```

The function span references a GCP load balancer span that's NOT in the trace.

### 6. Check Propagator Logs

View function logs for custom propagator activity:

```bash
gcloud functions logs read otel-test-function --gen2 --region=us-central1 --limit=50
```

Look for messages like:
```json
{
  "message": "[CloudRunPropagator] Injected backup header: x-original-traceparent=00-5719..."
}
```

```json
{
  "message": "[CloudRunPropagator] Found backup header, using original context: 00-5719..."
}
```

These confirm the custom propagator is working.

## What You're Testing

### The Problem

GCP Cloud Run's load balancer:
1. Creates intermediate spans (part of Google Cloud Trace)
2. Modifies the `traceparent` header to include its span ID
3. These LB spans are NOT exported to Last9
4. Result: Downstream spans reference missing parent IDs

### The Solution

Custom propagator that:
1. **On outgoing requests**: Copies `traceparent` → `x-original-traceparent`
2. **GCP load balancer**: Modifies `traceparent`, ignores `x-original-traceparent`
3. **On incoming requests**: Extracts from `x-original-traceparent` first
4. **Result**: Correct parent-child relationships preserved!

### Why It Matters

Last9's **trace-to-metrics logic** depends on proper span hierarchies to:
- Calculate accurate service dependencies
- Measure correct latency contributions
- Generate RED metrics per service
- Build service dependency graphs

## Troubleshooting

### No traces appearing in Last9

**Check credentials:**
```bash
# Verify secret exists
gcloud secrets describe last9-auth-header

# Check secret value
gcloud secrets versions access latest --secret=last9-auth-header
```

Should output: `Authorization=Basic <your-credentials>`

**Check function/service logs for errors:**
```bash
./deploy-and-test.sh logs
```

### Traces appear but hierarchy is still flat

**Verify custom propagator is deployed:**

1. Check function instrumentation:
```bash
gcloud functions describe otel-test-function --gen2 --region=us-central1 --format=json | jq '.buildConfig.source'
```

2. Redeploy to ensure latest code:
```bash
./deploy-and-test.sh function
./deploy-and-test.sh service
```

3. Ensure both services are using the propagator (check logs for propagator messages)

### Propagator messages not in logs

The custom propagator logs to `console.log`, so messages may be at INFO level.

**View all function logs:**
```bash
gcloud functions logs read otel-test-function --gen2 --region=us-central1 --limit=100 | grep -i propagator
```

**View all service logs:**
```bash
gcloud logging read "resource.type=cloud_run_revision AND resource.labels.service_name=otel-test-service" --limit=100 --format=json | jq -r '.[] | select(.jsonPayload.message | contains("Propagator"))'
```

## Advanced Testing

### Test Multiple Hops

Call the `/multi-hop` endpoint to test deeper call chains:

```bash
SERVICE_URL=$(cat .service_url)

curl -X POST "$SERVICE_URL/multi-hop" \
  -H "Content-Type: application/json" \
  -d '{"hops": 3, "data": {"test": "multi-hop"}}'
```

This creates a trace with multiple service ↔ function hops.

### Generate Sustained Load

```bash
for i in {1..50}; do
  curl -s "$(cat .service_url)/chain?name=LoadTest$i" > /dev/null &
  sleep 0.1
done
wait
```

Then analyze 50 traces in Last9 to verify consistency.

### Test Direct Function Calls

```bash
FUNCTION_URL=$(cat .function_url)

# Test 1: Simple call
curl "$FUNCTION_URL/?name=Test1"

# Test 2: POST with data
curl -X POST "$FUNCTION_URL/processData" \
  -H "Content-Type: application/json" \
  -d '{"key": "value", "items": [1,2,3]}'
```

These should create single-span traces (function as root).

When you're done testing:

```bash
./deploy-and-test.sh cleanup
```

This will prompt you to delete:
- Cloud Run Function
- Cloud Run Service

The Last9 credentials secret will remain (reusable).

To delete the secret:
```bash
gcloud secrets delete last9-auth-header
```

`★ Custom Propagator Validation ──────────────`

2. **Hierarchy**: Verify parent-child relationships in Last9
3. **TraceId**: Confirm all spans share the same TraceId
4. **ParentSpanId**: Ensure all references point to existing spans
5. **Metrics**: Last9 trace-to-metrics should now work correctly
